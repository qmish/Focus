## 🎓 Технические уроки

### Go Backend

#### ✅ Что работало хорошо
1. **Chi router** — лёгкий, быстрый, middleware
2. **GORM** — удобная ORM, но есть нюансы
3. **Wire dependency injection** — стоит использовать
4. **zap logger** — быстро, структурированно

#### ⚠️ Проблемы
1. **GORM AutoMigrate** — не для production, лучше golang-migrate
2. **sqlmock** — хрупкие тесты, лучше использовать testcontainers
3. **JWT claims** — внимательно проверять типы данных

#### 💡 Рекомендации
```go
// ✅ Хорошо
func NewUserRepository(db *gorm.DB) *UserRepository {
    return &UserRepository{db: db}
}

// ⚠️ Избегать
var db *gorm.DB // глобальная переменная
```

### React Frontend

#### ✅ Что работало хорошо
1. **Vite** — быстрая сборка, HMR
2. **Zustand** — проще Redux, достаточно для проекта
3. **React Router v6** — интуитивный роутинг

#### ⚠️ Проблемы
1. **Keycloak-js** — требует правильной настройки CORS
2. **Jitsi SDK** — ограниченная документация
3. **TypeScript strict mode** — ловит ошибки, но требует времени

#### 💡 Рекомендации
```typescript
// ✅ Хорошо
const { user, token } = useAuthStore()

// ⚠️ Избегать
const token = localStorage.getItem('token') // напрямую
```

### Kubernetes

#### ✅ Что работало хорошо
1. **HPA** — автоматическое масштабирование
2. **ServiceMonitor** — интеграция с Prometheus
3. **NetworkPolicies** — изоляция namespace

#### ⚠️ Проблемы
1. **Resource Quotas** — сложно подобрать правильные лимиты
2. **Ingress TLS** — cert-manager требует настройки DNS
3. **StatefulSet для БД** — лучше использовать managed PostgreSQL

#### 💡 Рекомендации
```yaml
# ✅ Хорошо
resources:
  requests:
    cpu: 250m
    memory: 256Mi
  limits:
    cpu: 1
    memory: 1Gi

# ⚠️ Избегать
resources: {} # без лимитов
```

### Безопасность

#### ✅ Что работало хорошо
1. **OIDC с PKCE** — защита от CSRF
2. **JWT с коротким TTL** — минимизация ущерба при утечке
3. **Security headers** — простая защита

#### ⚠️ Проблемы
1. **CORS** — сложно настроить правильно
2. **OWASP ZAP** — много ложных срабатываний
3. **Secret management** — Kubernetes Secrets не шифруются по умолчанию

#### 💡 Рекомендации
```go
// ✅ Хорошо
token, err := jwt.ParseWithClaims(tokenString, &Claims{}, func(token *jwt.Token) (interface{}, error) {
    return secret, nil
})

// ⚠️ Избегать
if token.Valid { // без проверки алгоритма
```

## 📋 Процессные уроки

### Планирование

#### ✅ Что работало хорошо
1. **Roadmap.md с чекбоксами** — видно прогресс
2. **Оценка по этапам** — реалистичные сроки
3. **Приоритизация** — сначала критический функционал

#### ⚠️ Проблемы
1. **Недооценка документации** — заняло больше времени
2. **Интеграции сложнее чем кажутся** — MS Exchange API
3. **Тесты требуют времени** — но окупаются

#### 💡 Рекомендации
- Закладывать +20% времени на интеграции
- Писать документацию параллельно с кодом
- Автоматизировать обновление Roadmap

### Разработка

#### ✅ Что работало хорошо
1. **TDD подход** — меньше багов
2. **Ежедневные коммиты** — виден прогресс
3. **Семантические версии** — понятно что изменилось

#### ⚠️ Проблемы
1. **Большой PR** — сложно ревьювить
2. **Отсутствие code owner** — кто отвечает за модуль
3. **Недостаточно примеров** — для сложных частей

#### 💡 Рекомендации
- PR < 400 строк
- Назначать code owners
- Добавлять примеры использования

### Тестирование

#### ✅ Что работало хорошо
1. **Unit тесты** — быстро, надёжно
2. **GitHub Actions** — автоматический запуск
3. **k6 для load тестов** — простые сценарии

#### ⚠️ Проблемы
1. **E2E тесты хрупкие** — зависят от окружения
2. **WebSocket тесты** — сложно мокировать
3. **Security scan** — требует ручного анализа

#### 💡 Рекомендации
- Использовать testcontainers для интеграционных тестов
- Иметь smoke тесты для production
- Запускать security scan еженедельно

### Документирование

#### ✅ Что работало хорошо
1. **C4 модель** — понятно архитектору и разработчику
2. **Диаграммы в Markdown** — версия в git
3. **API спецификация** — Swagger/OpenAPI

#### ⚠️ Проблемы
1. **Диаграммы устаревают** — нужно обновлять
2. **Слишком много деталей** — сложно найти главное
3. **Нет единого стиля** — разные авторы

#### 💡 Рекомендации
- Обновлять диаграммы при изменении кода
- Использовать шаблоны для документов
- Проводить review документации
